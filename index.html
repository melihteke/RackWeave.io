About RackWeave

RackWeave turns a datacenter rack & fiber design described in YAML into an interactive diagram, in the browser, with zero backend. Share the YAML the way engineers share code — the reader gets a first-class visual rendering rather than a screenshot.

Why YAML?

YAML is the language of network and infrastructure automation: Ansible, ContainerLab, intent files, design specs. RackWeave meets engineers where they already are. Drop a fiber design YAML unchanged and you get a diagram.

Why static?

Topology data is sensitive. RackWeave never sends your YAML anywhere — it’s a single static site that runs entirely in your browser. No account, no tracking, no backend.

YAML Documentation

RackWeave reads a single YAML document under the top-level key dc_fiber_design. It describes a datacenter site, its colocation zones, an admin (FCR) zone, and the fiber trunks running between racks.

1. Top-level structure

dc_fiber_design:
  site: SITE-A                   # short site code (string)
  site_metadata:                 # optional, free-form
    region: Region-1
    location: Example City

  default_power_feeds: [pdu-a, pdu-b]   # used when a rack omits its own

  colocations: [...]             # 1..N colocation zones (edge cabinets)
  admin_zone:    {...}           # optional center "FCR" zone
  fiber_connections_between_racks: {...}

2. colocations[] — edge zones

Each colocation is a zone of edge racks. The first colocation is rendered on the left, the second on the right, alternating after that.

colocations:
  - id: 1
    name: ZONE-A                 # zone label shown in the diagram
    racks:
      - name: SITE-A.ZONE-A.RACK-01    # required, must be unique
        power_feeds: [pdu-a, pdu-b]    # optional, overrides default
        devices:                       # optional list of device hostnames
          - SITE-A-ZA-SW-01
          - SITE-A-ZA-SW-02

3. admin_zone — Fiber Cross-Connect Room

Optional. Rendered in the center as amber FCR cabinets.

admin_zone:
  name: CORE
  racks:
    - name: SITE-A.CORE.FCR-01
      role: Fiber-Cross-Connect-Room   # optional sub-label
      power_feeds: [pdu-a, pdu-b]

4. fiber_connections_between_racks — trunks

Connections between rack nodes. Accepts either a flat list, or named groups (recommended) so you can color/filter logically related trunks.

fiber_connections_between_racks:
  zone_a_connections:
    - { from: SITE-A.ZONE-A.RACK-01, to: SITE-A.CORE.FCR-01, strands: 24, type: SMF }
    - { from: SITE-A.ZONE-A.RACK-02, to: SITE-A.CORE.FCR-02, strands: 12, type: MMF }
  zone_b_connections:
    - { from: SITE-A.ZONE-B.RACK-01, to: SITE-A.CORE.FCR-01, strands: 48, type: SMF }
FieldRequiredNotes
fromyesSource rack name
toyesDestination rack name
strandsrecommendedNumber of strands; shown on the link mid-label

5. Fiber types — SMF vs MMF

The type field is free-form, but RackWeave recognizes the two industry-standard values:

ValueMeaningTypical use
SMFSingle-Mode Fiber (OS1/OS2, 9/125 µm)Long reach (≥ ~500 m), DCI, spine ↔ FCR, dark fiber
MMFMulti-Mode Fiber (OM3/OM4/OM5, 50/125 µm)Intra-row / short reach (< ~300 m), ToR ↔ leaf, low-cost optics

The value appears in the strand label (e.g. 24-SMF, 12-MMF) and in tooltips. You can use any string (e.g. OM4, OS2, AOC) — it will be displayed verbatim.

6. Tips

  • Rack name values must be globally unique — they are the link join keys.
  • Strand labels (24-SMF, 12-MMF, …) are draggable on the canvas — reposition them anywhere if a link is busy.
  • Hit Auto-Layout to discard manual rack positions and recompute.
  • Use Examples → 03 · Large multi-zone for an end-to-end reference YAML.
